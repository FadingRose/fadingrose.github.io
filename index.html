简报

这是一个关于Go-callvis安装的踩坑记录，虽然最终的原因已经和这个插件没关系了（

发生了什么？

Go-callvis本身是一个Go实现的生成函数调用图的插件，根据Github上的Quick Start指引，
源代码下面有一个Makefile文件，借助这个文件来完成安装

要求在go-callvis的源代码文件夹下执行 make install，此时的文件结构树有个这个东西

奇怪的地方出现了，执行完后没有任何反应，go-callvis在zsh里面也没有生效。

开始拆弹

我的第一反应是这个包没有正常装上，毕竟之前装的其他包命令都可以调用，

但是当在Google上检索了多个类似go-callvis cann’t found的关键词后，发现全世界的人
都没有这个症状，因此基本排除是包本身有问题。

那么接下来就要看看安装中到底发生了什么，这需要从Makefile入手

这就要看明白 make install 到底是在干嘛，所以决定先看看Makefile文件

执行 code -r Makefile 拿VSCODE看看

多说一句 code 命令本身没有加入zsh 需要在vscode的命令行（cmd+shift+P）输入 shell 启用里面的 “Shell Command：Install code command to PATH” 之后就可以愉快的用code来指定用vscode打开文本了

Makefile文件节选如下：

虽然前面还有一大串花里胡哨的变量定义，但是很快发现 make install 实际上只是执行了 go install 而已

这个prefix很好理解，比如你执行了 make all 那么就会执行 all:下面的指令

那么问题来了，这个插件到底有没有装上呢？还是不知道

还好强大的Google给我找到了一个半残的网页（这个网页除了文本内容其他已经加载不出来了）

在一大串字里行间里发现了一点关键线索： go-callvis 和 $GOROOT/go/bin。

随后再看看自己的文件目录里有没有这个东西，结果在GOROOT/go/bin 下找到了一个Unix可执行文件名字叫 go-callvis ，这说明插件已经装上了。

那么到了这里拆弹已经完成了90%，因为几乎已经可以是zsh没有找到这个文件夹在哪，也就是说环境变量出了问题。

检查一下自己的变量 cat ~/.zshrc

节选如下：

果然一眼丁真鉴定为寄，PATH路径压根没有指向一个正确的文件夹，那zsh当然找不到 go-callvis在哪，简单修改一下：

export PATH="$HOME/go/bin:$PATH"

再source ~/.zshrc重新生效一下

然后回到zsh检查一下

Boom has been defused～

接下来就可以愉快的看调用图了

下面是一点奇怪的地方

之前在搭建go的调试环境的时候装了dlv调试桥，那个时候环境变量也是错的，但为什么可以找到正确的路径呢？

一个猜想是vscode内建的command line有额外的搜索规则，然后顺便把路径给补上了

不过这也只是猜想，这包的管理和安装其实就没有弄明白过，不过没有关系，能用就行
